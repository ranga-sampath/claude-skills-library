**Drop type: Stateful mid-path discard.** The drop pattern — same stream, same direction, original + retransmit both dropped, sibling stream unaffected — is consistent with a stateful policy at a firewall, load balancer, or middlebox. Stateless random packet loss would not consistently target one stream while leaving a parallel stream from the same source IP to the same destination port completely clean.

## Root Cause Analysis

The six missing packets (3 original + 3 retransmit) on stream 1 while stream 0 is unaffected cannot be explained by physical link loss or general congestion — both would affect both streams proportionally. The selective nature of the drop points to a policy or state-based middlebox decision applied to stream 1's 5-tuple (10.0.1.10:52002 → 10.0.2.20:8080) but not to stream 0's 5-tuple (10.0.1.10:52001 → 10.0.2.20:8080).

Candidate causes:
- **Firewall ACL hit:** A rule matching stream 1's source port (52002) or a port range that excludes 52001 is dropping packets for that stream silently (no ICMP Unreachable visible — not present in the capture).
- **Connection tracking table full or corrupted:** The stateful firewall lost track of stream 1's connection state and began dropping non-SYN packets as untracked. Stream 0 was established earlier and its state entry is intact.
- **Load balancer session affinity failure:** If a load balancer sits in the path, stream 1 may have been assigned to a backend node that is down or rejecting traffic, while stream 0 is assigned to a healthy backend.
- **QoS policy or rate limiter:** A per-flow rate limiter with a threshold between the two stream rates is silectively dropping stream 1 once it exceeds a byte or packet quota.

The absence of ICMP Unreachable or RST from the path (no ICMP section in either capture) means the dropping device is silently discarding packets — typical of a firewall DROP rule (vs REJECT).

## Remediation

**Immediate — Identify the dropping device:**
- Run `traceroute -T -p 8080 10.0.2.20` from 10.0.1.10 — compare the hop count for stream 1's port vs stream 0's port.
- Capture at an intermediate hop (e.g., the default gateway of 10.0.1.10) and run the same endpoint-correlation analysis: if the drop moves to appear before the intermediate hop, the dropping device is upstream of it.

**Firewall — Check for ACL or connection tracking issues:**
- On the firewall between 10.0.1.0/24 and 10.0.2.0/24: `show conn | grep 10.0.1.10` (ASA) or `conntrack -L | grep 52002` (Linux netfilter) — verify stream 1 has an active state entry.
- Review ACL rules for the 10.0.1.10 → 10.0.2.20:8080 flow: `show access-list` or `iptables -L -n -v`.
- If connection tracking is the cause: `conntrack -D -s 10.0.1.10 -d 10.0.2.20 --dport 8080` to flush the stale entry, then retransmit.

**Load balancer — Check session assignment:**
- Verify 10.0.2.20:8080 is a VIP, not a real host. If it is a VIP, check session tables: `show persistence` (F5) or equivalent.
- Confirm all backend pool members are healthy.

**Validation:**
- After applying a fix, re-run the endpoint-correlation capture pair for stream 1. Target result: retransmissions = 0 at source, packet counts match at source and destination.
