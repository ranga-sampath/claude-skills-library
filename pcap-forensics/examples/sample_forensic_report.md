## Root Cause Analysis

**ARP Spoofing (CRITICAL)**
Two distinct MAC addresses are asserting the same IP (10.0.0.5) via ARP replies. Per RFC 826, ARP is stateless and unauthenticated — any host can broadcast an ARP reply claiming any IP-MAC mapping, and receiving hosts update their cache accordingly. The attacker at `ff:ee:dd:cc:bb:aa` is intermittently winning the ARP cache race, causing traffic destined for 10.0.0.5 to be forwarded to the attacker instead of the legitimate host at `aa:bb:cc:dd:ee:05`. This enables man-in-the-middle interception or denial of service for that IP. The single gratuitous ARP is consistent with the attacker periodically refreshing the poisoned mapping across all hosts on the segment.

**DGA Malware Beaconing (CRITICAL)**
Five NXDOMAIN responses share an identical structural pattern: 8-character pseudo-random alphanumeric subdomains under the fixed parent `evil-c2.com`. This is the canonical DGA signature — malware derives a deterministic sequence of candidate hostnames from a seed (often the current date) and queries each in turn until one resolves to an active C2 server. The NXDOMAIN responses confirm the C2 infrastructure is currently offline or the seed hasn't rotated yet, but the infected host is present and actively executing the beaconing loop. The parent domain `evil-c2.com` is fixed and blockable at DNS and firewall. The threat is the infected host, not the resolver — 10.0.0.53 is correctly returning NXDOMAIN.

**Host 10.0.0.99 Confirmed Down (HIGH)**
ARP unanswered requests establish that 10.0.0.99 is not responding to L2 broadcast — it is either powered off, physically disconnected, or on the wrong VLAN. Router 10.0.0.254 independently confirms this at L3: ICMP Host Unreachable Code 1 means the router has a route to the destination network but cannot ARP-resolve the specific host. Per RFC 792, this is the router's notification that it reached the correct network but the host itself is unreachable at the data-link layer. The `unreachable_dst: 10.0.0.99` field (extracted from the embedded inner IP header in the ICMP payload) confirms 10.0.0.99 is the actual target — not to be confused with `dst: 10.0.0.1`, which is merely the recipient of the error notification.

**TCP Stream 1 Degradation (MEDIUM)**
Stream 1 shows 5 retransmissions and 210ms median ACK RTT while stream 0 to the same destination (10.0.0.2:80) runs cleanly at 5ms. Degradation on one stream while a parallel stream to the same server is healthy rules out network-wide packet loss. The pattern — high RTT, retransmissions, RST teardown — points to application-layer slowness on the server handling that specific connection: slow query processing, thread pool exhaustion, or resource contention causing delayed ACKs, triggering RTO-based retransmissions, and ultimately connection abandonment via RST.

## Remediation

**ARP Spoofing:**
- Locate and disconnect the attacker: `show mac address-table address ff:ee:dd:cc:bb:aa` on the access switch → identify and shut the port.
- Enable Dynamic ARP Inspection: `ip arp inspection vlan <vlan_id>` on the switch.
- Flush poisoned ARP caches on hosts: `arp -d 10.0.0.5 && ip neigh flush dev eth0`

**DGA Malware Beaconing:**
- Identify the infected host from DNS query logs on 10.0.0.53 — find the source IP issuing queries for `*.evil-c2.com`.
- Isolate immediately: disconnect the host before the DGA resolves to an active C2 server.
- Block at DNS (Response Policy Zone) and firewall: `iptables -A FORWARD -p udp --dport 53 -m string --algo bm --string "evil-c2.com" -j DROP`
- Reimage after forensic capture: `ss -tnp`, `netstat -an`, memory image if required.

**Host 10.0.0.99 Unreachable:**
- Confirm physical/virtual status. If decommissioned: remove DNS entries and static routes.
- If it should be active: verify VLAN assignment and subnet mask on the connecting switch port (`show interface`, `show mac address-table`).

**SERVFAIL — broken.internal:**
- Trace the failure: `dig +trace broken.internal @10.0.0.53`
- Check authoritative server health: `systemctl status named`
- Validate zone file: `named-checkzone broken.internal /etc/bind/zones/broken.internal.zone`
- Restart if valid: `systemctl restart named`

**TCP Stream 1 Retransmissions:**
- The ICMP finding (10.0.0.99) is unrelated — stream 1 goes to 10.0.0.2. Investigate independently.
- On 10.0.0.2: `netstat -s | grep retrans`, check application logs for slow requests during the retransmission window, review CPU/memory.
- Check duplex: `ethtool eth0 | grep -i duplex` on both 10.0.0.1 and 10.0.0.2.

**TTL Exceeded — 172.16.0.1 → 192.168.99.1:**
- Check routing table on 172.16.0.1: `show ip route 192.168.99.1` — look for a circular or recursive route.
- Trace the loop: `traceroute 192.168.99.1` from 10.0.0.1.

**DNS Slow Queries:**
- Identify the forwarder: `dig +trace slow0.remote.com @10.0.0.53` — find where latency is introduced.
- Check connectivity to the authoritative NS for `remote.com`: `dig NS remote.com` then `ping <ns_ip>`.
