## Root Cause Analysis

**TCP Retransmissions (Resolved):** The retransmission pattern in capture A — 5 retransmissions with 200ms inter-segment delays and 210ms ACK RTT — indicates the server at 10.0.0.2:80 was ACK-ing slowly, causing the sender's retransmission timer (RTO) to fire. The delta p95 of 200ms aligns with a 200ms RTO floor. In capture B, the same server responds at 5ms — the server-side condition (likely CPU, thread, or queue saturation) had cleared. Network-layer loss is ruled out: ICMP RTT is identical in both captures (200ms baseline, no added delay), confirming the path itself was not degraded.

**ARP Resolution (Resolved):** ARP is stateless — a host either responds to broadcasts for its IP or it doesn't. The transition from 0 replies to 5 replies is binary: 10.0.0.99 was not present in capture A and is present in capture B. The cause was the host's absence from the network, not an ARP configuration issue.

**DNS Slow Queries (Persistent):** 10.0.0.53 resolves `example.com` queries via an upstream forwarder. The 500ms latency, consistent across both captures, points to a fixed upstream latency characteristic — the authoritative NS for `example.com` is slow to respond to this resolver. Since 15ms queries go through the same resolver, the bottleneck is the upstream path to `example.com`'s NS, not 10.0.0.53 itself.

## Remediation

**DNS Slow Queries (Open):**
- Identify the upstream resolver path: `dig +trace host0.example.com @10.0.0.53` — find which hop introduces the 500ms delay.
- Check if 10.0.0.53 has a forwarder configured for `example.com`: inspect `/etc/bind/named.conf` or equivalent for zone-specific forwarder entries.
- If the upstream NS is geographically distant, consider adding a recursive NS closer to the network, or increasing the TTL on `example.com` records to reduce query frequency.
- If 10.0.0.53 is a corporate resolver with a forwarder chain, check whether the intermediate forwarder is adding latency: `dig host0.example.com @<intermediate_forwarder_ip>`.
